# Mem AI 常见坑与处理

## 1. 先用 `memctl`，不要默认手写 `curl`

优先执行 `scripts/memctl`。
只有在以下情况才直接打底层 API：
- 需要 `mem-it`
- 需要 Collection 的创建 / 搜索 / 删除
- 需要调试原始响应

## 2. 默认不要写三反引号代码块

保存到 Mem 的正文默认使用普通标题、列表、短段落和行内命令。
避免把内容写成 ` ```bash `、` ```text ` 这类代码块，以免在预览或二次转存时出现转义字符，影响阅读与搜索。

## 3. 更新不是独立接口，而是同 `id` 覆盖写回

Mem 没有单独的 update endpoint。
更新流程应为：
1. `read` 读取旧内容
2. 根据场景决定 `replace` / `append` / `prepend`
3. 重新 `POST /v2/notes` 并带上已有 `id`

多行追加或替换时，优先使用 `--file`，不要把 `\n` 当普通字符串直接塞进命令。

## 4. 搜索默认走“语义优先”，不要机械堆关键词

优先写成一句自然语言，必要时再补 2~4 个核心词。
只有在结果太杂或太少时再收窄 / 放宽条件。

## 5. 标题首先服务检索，不追求短促好看

优先写清：
- 主题
- 用途 / 作用
- 场景 / 任务词
- 必要时补一个限制词

推荐模板：
- `主题：作用/用途`
- `主题：作用/用途 + 场景`
- `工具/对象：适用任务 + 关键限制`

## 6. 结果很多时，不要把全文一股脑倒给用户

默认先：
- 返回 3~5 条最相关结果
- 每条给标题、id、更新时间、1 句摘要
- 让用户挑一条再 `read`

只有用户明确要全文，或命中极少时，再展示完整内容。

## 7. 更新笔记时，默认保留旧 Collection

使用 `memctl update` 时，默认保持现有 `collection_ids`。
只有用户明确指定新的 Collection，才覆盖归属。

## 8. 删除属于高风险动作

删除前至少确认以下之一：
- 用户给了明确 `note_id`
- 用户从候选结果里明确选中了一条

如果只是模糊说“删掉那个”，先列候选，别直接删。

## 9. 写入后要给用户“有价值”的反馈

不要只说“已保存”。优先补充一个最有价值的信息：
- 保存到哪个 Collection
- 标题是什么
- 如果是整理类内容，概括提炼出了什么
- 如果是搜索，返回了多少条和最相关的几条

## 10. 用户个人语境的专有名词，先补一次知识库搜索

如果输入里出现明显属于用户个人语境的专有名词，并且它会影响理解、归类、关联或检索结果，先去用户知识库搜索一次再继续。

优先顺序：
1. 先查会话可见的长期/短期记忆
2. 如果仍不足，再查用户的 Mem AI 知识库
3. 再决定是回答、保存、更新、归类还是继续追问

典型例子：
- 人名：`孙总`、`王老师`
- 内部简称：团队黑话、项目代号、公司内部叫法
- 只在用户上下文里成立的指代

默认采用保守模式：
- 只有当这个词明显属于用户个人语境，且不查就容易误解时，才先搜索
- 不要因为普通通用名词频繁触发检索

## 11. 遇到这些场景，优先走任务级工作流

- 原始长文本 / 网页 / 会议记录 → 先看 `references/mem-it.md`
- 搜索旧方案 / 历史决策 → 先看 `references/search.md`
- 追加进展 / 更新状态 → 先 `read` 再 `update`
- 需要用户可感知的话术 → 看 `references/response-patterns.md`
